# Implementation Plan

- [x] 1. プロジェクト初期化と設定管理
- [x] 1.1 Python プロジェクトの初期化
  - uv でプロジェクトを作成し、Python 3.12 をランタイムに指定する
  - discord.py、openai、python-dotenv を依存関係として追加する
  - `.gitignore` に `.env`、`__pycache__/`、`.venv/` を含める
  - `.env.example` を作成し、必要な環境変数名を列挙する（値は空）
  - _Requirements: 7.1, 8.1_

- [x] 1.2 設定管理モジュールの実装
  - 環境変数から全ての接続情報を読み込む Config を実装する
  - Discord トークン、Azure AI Search のエンドポイント・API キー・インデックス名、Azure OpenAI のエンドポイント・API キー・デプロイメント名・API バージョンを管理する
  - 必須項目が欠けている場合は起動時にエラーを送出して停止する
  - 設定値は読み取り専用とし、起動後に変更できないようにする
  - _Requirements: 7.1, 7.2, 8.1_
  - _Contracts: Config State_

- [x] 2. Azure AI Search 検索サービスの実装
- [x] 2.1 (P) 検索サービスの基本実装
  - Azure AI Search の REST API に対して検索クエリを送信し、関連ドキュメントを取得する機能を実装する
  - 質問文をそのまま検索クエリとして使用し、BM25（キーワード検索）で関連度の高い上位ドキュメントを取得する
  - 検索結果からドキュメントの ID、タイトル、本文、ソース、スコアを抽出してデータ構造に変換する
  - 認証には環境変数から読み込んだ API キーを使用する
  - _Requirements: 2.1, 2.2, 2.4_
  - _Contracts: SearchService Service, SearchService API_

- [x] 2.2 (P) 検索結果なしと検索エラーのハンドリング
  - 検索結果が 0 件の場合、空のリストを返し、呼び出し元が適切に判断できるようにする
  - Azure AI Search への接続失敗（401, 403, 429, 503 等）をキャッチし、検索固有のエラーとして上位に伝播する
  - エラー発生時にエンドポイント、ステータスコード、レスポンス情報を開発者向けログに記録する（API キーはログに含めない）
  - _Requirements: 2.3, 5.1, 5.4, 6.4_

- [x] 3. AI 回答生成サービスの実装
- [x] 3.1 (P) 回答生成サービスの基本実装
  - 検索結果をコンテキストとしてシステムプロンプトに埋め込み、Azure AI Foundry の gpt-4o-mini で回答を生成する機能を実装する
  - システムプロンプトに anna キャラクターの口調指示（親しみやすく、丁寧で、軽くかわいげのある口調）と回答ルール（検索結果のみに基づく、推測しない）を含める
  - 検索結果の各ドキュメントのタイトルと本文をコンテキストとしてフォーマットする
  - 回答テキスト、参照元タイトルのリスト、十分なコンテキストがあったかどうかのフラグを返す
  - temperature=0.3、max_tokens=800 を設定する
  - _Requirements: 3.1, 3.2, 4.1, 4.2, 4.3, 4.4_
  - _Contracts: AIService Service_

- [x] 3.2 (P) 検索結果なし時の回答と生成エラーのハンドリング
  - 検索結果が空の場合、「部内資料には見当たらなかった」旨を anna の口調で回答するようプロンプトを制御する
  - 回答に根拠となる資料がある場合は「参考: 〇〇」の形式で参照元タイトルを末尾に付与する
  - Azure AI Foundry への API 呼び出し失敗（429, 500, 503 等）をキャッチし、生成固有のエラーとして上位に伝播する
  - エラー詳細と処理時間を開発者向けログに記録する
  - _Requirements: 3.3, 3.4, 5.2, 5.4_

- [x] 4. Discord Bot とスラッシュコマンドの実装
- [x] 4.1 Bot の起動とコマンド登録
  - discord.py を使って Bot を起動し、`/anna` スラッシュコマンドを登録する
  - コマンドには `question`（質問内容）というテキスト入力パラメータを設ける
  - Bot 起動時に Config の読み込みとバリデーションを行い、設定不備があれば起動を中止する
  - 開発中は Guild コマンドとして登録し即時反映させる（本番では Global に切り替え可能にする）
  - _Requirements: 1.1, 7.2, 8.1_

- [x] 4.2 質問応答のオーケストレーション
  - `/anna` コマンド受信時、最初に Deferred Response（考え中）を返し、3 秒以内の応答期限を満たす
  - 検索サービスで質問文を検索し、結果を回答生成サービスに渡して回答を生成する一連のフローを実装する
  - 生成された回答を followup メッセージとして送信し、参照元がある場合は回答に含める
  - 回答が 2000 文字を超える場合は Embed を使用するか末尾を省略する
  - リクエスト受信時にタイムスタンプとユーザー識別子をログに記録し、回答完了時に処理時間と成功/失敗をログに記録する
  - _Requirements: 1.2, 1.3, 3.4, 6.1, 6.2_

- [x] 4.3 エラーハンドリングとユーザー向けメッセージ
  - 検索エラー、生成エラー、予期しないエラーをそれぞれキャッチし、anna の口調でユーザーにわかりやすいエラーメッセージを返す
  - エラーメッセージに内部システムの詳細（URL、IP、スタックトレース）を含めない
  - 全てのエラーで詳細情報（スタックトレース、リクエスト情報、タイムスタンプ）を開発者向けログに記録する
  - ユーザー向けメッセージと開発者向けログを明確に分離する
  - _Requirements: 5.1, 5.2, 5.3, 5.4, 6.3, 6.4, 7.4_

- [x] 5. GitHub Actions によるインデックス自動更新
- [x] 5.1 (P) インデクサーワークフローの実装
  - 部の GitHub Organization の private リポジトリに push されたときに自動実行される GitHub Actions ワークフローを作成する
  - main ブランチへの push をトリガーとし、リポジトリ内の Markdown ファイルを対象にする
  - 各 Markdown ファイルからタイトル（最初の `#` 見出し）、本文、ファイルパス（ソース）を抽出する
  - 長文ドキュメントは見出し（`##`）単位でチャンクに分割し、各チャンクに元ドキュメントのタイトルとソースを保持する
  - ファイルパスからドキュメント ID を生成し、Azure AI Search の Push API で `mergeOrUpload` アクションを使ってインデックスに投入する
  - Azure の認証情報は GitHub Secrets から読み込む
  - _Requirements: 2.5, 7.1, 8.2_
  - _Contracts: IndexerWorkflow Batch_
